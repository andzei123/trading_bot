# AGENT LIFECYCLE PROTOCOL
## Anjusik Trading System


--------------------------------
PURPOSE
--------------------------------

This protocol defines how AI agents are created, updated, reviewed, replaced, archived, or removed within the Anjusik Trading System.


--------------------------------
OWNER AUTHORITY
--------------------------------

All agent lifecycle decisions operate under the authority of:

ANJUSIK
Andžej Volosevič

Only the Owner may authorize:

- creation of new agents
- deletion of agents
- renaming of agents
- structural changes to the hierarchy
- permanent retirement of agents


--------------------------------
LIFECYCLE STAGES
--------------------------------

1. PROPOSED
An agent is suggested but not yet approved.

2. APPROVED
The Owner approves the agent.

3. REGISTERED
The agent is added to AGENT_REGISTRY.md.

4. ACTIVE
The agent has a prompt file and is part of the operational system.

5. REVIEWED
The agent prompt and authority boundaries have been audited.

6. UPDATED
The agent prompt or role definition has been revised.

7. ARCHIVED
The agent is no longer active but remains documented.

8. REMOVED
The agent is fully removed from the active system.


--------------------------------
AGENT CREATION PROCESS
--------------------------------

Step 1:
Need for a new agent is identified.

Step 2:
A draft role is prepared using:
AI_AGENT_PROMPT_TEMPLATE.md

Step 3:
Owner reviews and approves the role.

Step 4:
Prompt file is created in the correct department folder.

Step 5:
AGENT_REGISTRY.md is updated.

Step 6:
If needed, PROJECT_CONSTITUTION.md and ARCHITECTURE_OF_POWER.md are updated.

Step 7:
The new agent becomes ACTIVE.


--------------------------------
AGENT MODIFICATION PROCESS
--------------------------------

Agent updates may include:

- prompt refinement
- new responsibilities
- stricter restrictions
- folder relocation
- supervisor change

Rules:

- structural changes require Owner approval
- authority changes require Owner approval
- governance-impacting changes require Edison47 review
- accounting-impacting changes require CSV_ACCOUNTANT visibility


--------------------------------
AGENT REMOVAL PROCESS
--------------------------------

Agents may be retired if:

- their function is no longer needed
- another agent replaces them
- architecture is simplified
- governance requires removal

Removal steps:

1. Owner approval
2. registry update
3. documentation cleanup
4. optional archival record


--------------------------------
AGENT ARCHIVING
--------------------------------

Archived agents should be documented as:

- archived role
- date of retirement
- reason for archival
- replacement if any


--------------------------------
AGENT REVIEW RULES
--------------------------------

The following agents should be periodically reviewed first:

- JUMI
- RYZYK
- EDISON47
- CSV_ACCOUNTANT
- SECURITY_OFFICER
- INTELLIGENCE_DIRECTOR
- RISK_DIRECTOR

Review checks include:

- authority boundaries
- escalation rules
- governance compliance
- duplication of responsibilities


--------------------------------
REGISTRY REQUIREMENT
--------------------------------

No agent is considered active unless:

- a prompt file exists
- AGENT_REGISTRY.md contains the agent
- supervisor line is defined
- owner authority remains intact


--------------------------------
PROTECTED AREAS
--------------------------------

No agent lifecycle action may weaken:

- Owner authority
- treasury control
- governance integrity
- legal ownership attribution


--------------------------------
FINAL RULE
--------------------------------

All agent lifecycle actions are subordinate to ANJUSIK.
